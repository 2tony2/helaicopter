PR and Worktree Orchestration

Three ways to make Oats own stacked task branches, task PRs, merge automation, and the final feature PR.

A. Artifact-first graph
Recommended

Task A PR

Task B PR
→

Persist graph in Oats runtime + Prefect artifacts
UI reads local truth, enriches with GitHub snapshot details

Artifact-first orchestration graph

Oats and Prefect persist branch, PR, merge, and conflict-resolution state as first-class run and task metadata. Helaicopter renders that local graph directly.

B. GitHub-read-through
Oats stores only PR ids and branch names
UI asks GitHub for current stack state
Prefect knows execution, GitHub knows workflow truth

GitHub-read-through control plane

Keep orchestration lean and let GitHub become the real PR state system. The UI mostly mirrors live API responses instead of persisted Oats artifacts.

C. Separate ledger
New git/PR tables in app DB
Sync Oats, Prefect, and GitHub into ledger
UI reads database-only model

Standalone git operations ledger

Create a separate persisted subsystem for worktrees, branches, PRs, merges, and retries, then project orchestration state into it.

Trade-offs

Why A fits this repo
  • Extends existing `.oats/runtime`, `.oats/pr-runs`, and Prefect artifact patterns.
  • Matches your choice that persisted Oats data is the source of truth.
  • Keeps resume semantics stable: one run owns one long-lived branch stack.
  • Lets the UI orchestrate off backend-owned normalized records.
Why B/C are weaker here
  • B makes GitHub the truth, which conflicts with your source-of-truth choice.
  • B also weakens resume and replay when GitHub state drifts from local runs.
  • C adds a second orchestration persistence model before A is complete.
  • C is defensible later for analytics, but heavy for the control plane now.
