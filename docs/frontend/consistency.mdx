## URL and API conventions

### Never hard-code API paths

Always use the builder functions from `src/lib/client/endpoints.ts`:

```typescript
import { conversationsUrl, analyticsUrl } from '@/lib/client/endpoints'

// Correct
const url = conversationsUrl(project, days)

// Wrong — don't do this
const url = `http://localhost:30000/conversations?project=${project}`
```

### Backend origin

Configured via `NEXT_PUBLIC_API_BASE_URL`. When unset on localhost, the client automatically falls back to `:30000`.

## Data fetching patterns

### Use SWR hooks

All data fetching goes through the hooks in `src/hooks/`:

```typescript
import { useConversations, useAnalytics } from '@/hooks/use-conversations'

function MyComponent() {
  const { data, error, isLoading } = useConversations(project, days)
  // ...
}
```

### Default refresh intervals

| Hook | Interval |
|------|----------|
| Conversations | 3 seconds |
| Analytics | 15 seconds |
| Database status | 5 minutes |

### Mutations

Write operations use the functions in `src/lib/client/mutations.ts`:

```typescript
import { refreshDatabase, createEvaluationPrompt } from '@/lib/client/mutations'

await refreshDatabase({ fullRefresh: true })
```

## Response normalization

The backend returns `snake_case` fields. The normalizer in `src/lib/client/normalize.ts` maps them to TypeScript types defined in `src/lib/types.ts`.

During the migration period, the normalizer accepts both casings. For new endpoints, prefer `snake_case` from the backend and let the normalizer handle the mapping.

## Component patterns

### UI primitives

Use the existing shadcn-style components from `src/components/ui/`:

- `Card`, `CardHeader`, `CardTitle`, `CardContent` — content containers
- `Badge` — status indicators, tags
- `Tabs`, `TabsList`, `TabsTrigger`, `TabsContent` — tabbed interfaces
- `Button` — actions with variants (default, outline, ghost)
- `Input`, `Textarea` — form inputs
- `Skeleton` — loading placeholders
- `Dialog` — modal dialogs
- `ScrollArea` — scrollable containers
- `Tooltip` — hover hints

### Page structure

Every page follows this pattern:

```typescript
import { PageHeader } from '@/components/layout/page-header'
import { Container } from '@/components/layout/container'

export default function MyPage() {
  return (
    <>
      <PageHeader title="Page Title" description="What this page does" />
      {/* Page content */}
    </>
  )
}
```

### Feature component organization

The codebase is migrating to an explicit layer structure. New code should follow these conventions:

- `src/features/<domain>/components/` — feature-owned components (e.g., `src/features/plans/components/`)
- `src/shared/ui/` — reusable UI primitives (shadcn-style components)
- `src/shared/layout/` — shared layout pieces (sidebar, headers)
- `src/views/<domain>/` — page-level composition components

Legacy `src/components/` paths remain as compatibility shims and continue to work, but new additions should go into the layer structure above.

Current domain locations:
- `src/components/analytics/` — dashboard charts, KPI cards (not yet migrated)
- `src/components/conversation/` — message display, DAG views (not yet migrated)
- `src/components/orchestration/` — orchestration panels (not yet migrated)
- `src/components/databases/` — database status and inspection (not yet migrated)
- `src/features/plans/components/` — plan viewing (migrated)
- `src/components/evaluations/` — prompt management (not yet migrated)

## Type system

### Core types (`src/lib/types.ts`)

Key interfaces:

- `ConversationSummary` — list-level session data
- `ProcessedConversation` — full detail with messages
- `ProcessedMessage` / `DisplayBlock` — message rendering
- `AnalyticsData` — dashboard data structure
- `ConversationDag` / `ConversationDagNode` — DAG visualization
- `DatabaseStatus` — database health information

### Naming conventions

| Layer | Convention | Example |
|-------|-----------|---------|
| Backend schemas | `*Record`, `*Response` | `ConversationRecord`, `AnalyticsDataResponse` |
| Frontend types | Descriptive, no suffix | `ConversationSummary`, `AnalyticsData` |
| Hooks | `use*` | `useConversations`, `useDatabaseStatus` |
| URL builders | `*Url` | `conversationsUrl`, `analyticsUrl` |

Frontend display models in `types.ts` are stable and should not mirror Pydantic model names directly.

## Pricing and costs

Token costs are calculated in `src/lib/pricing.ts` using pricing tables from `src/lib/constants.ts`. The `getPricing(model)` function uses prefix matching — `claude-sonnet-4-5-20250514` matches the `claude-sonnet-4-5` pricing tier.

Pricing covers both Claude models (Opus, Sonnet, Haiku) and OpenAI models (GPT-5.x, o3, o4-mini) with per-token rates for input, output, cache read, and cache write.
