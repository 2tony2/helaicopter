## System design

Helaicopter is a two-process local application:

| Layer | Tech | Port | Responsibility |
|-------|------|------|----------------|
| Frontend | Next.js 16, React 19, SWR | 3000 | UI rendering, client-side caching, chart visualization |
| Backend | FastAPI, SQLAlchemy 2, Pydantic v2 | 30000 | Data parsing, analytics, database management |

The frontend never reads the filesystem directly. All conversation parsing, token aggregation, and database operations happen in the backend. The frontend consumes a typed JSON API and uses SWR for client-side caching with configurable refresh intervals.

## Data flow

```
~/.claude/projects/.../*.jsonl ──┐
                                 ├──▶ FastAPI Backend ──▶ JSON API ──▶ Next.js Frontend
~/.codex/sessions/.../*.jsonl ───┘         │
                                           ▼
                                   SQLite (OLTP)
                                   DuckDB (OLAP)
```

### Read path (hot)
1. Frontend SWR hook calls a backend endpoint (e.g., `GET /conversations`)
2. Backend checks in-process LRU cache
3. On miss: reads from SQLite/DuckDB for persisted data, or parses JSONL files directly
4. Response is serialized through Pydantic schemas
5. Frontend normalizer maps to TypeScript types
6. SWR caches the response client-side

### Write path (refresh)
1. User triggers `POST /databases/refresh` (or `npm run db:refresh`)
2. Backend scans `~/.claude/` and `~/.codex/` for conversation files
3. Streaming JSONL parser processes each file (handles 200MB+ without loading into memory)
4. Records written to SQLite (conversations, messages, blocks, plans, tasks)
5. Analytics materialized in DuckDB (time-series, aggregates)
6. All backend caches invalidated, SQLAlchemy engine disposed
7. Schema docs regenerated under `public/database-schemas/`

## Backend architecture

The backend follows **hexagonal architecture** (ports and adapters):

```
┌─────────────────────────────────────────────┐
│                  Routers                     │  ← HTTP layer (FastAPI APIRouter)
├─────────────────────────────────────────────┤
│               Application                    │  ← Business logic, joins, shaping
├─────────────────────────────────────────────┤
│                 Schemas                      │  ← Pydantic v2 response models
├─────────────────────────────────────────────┤
│            Ports (interfaces)                │  ← Protocol classes for DI
├─────────────────────────────────────────────┤
│          Adapters (implementations)          │  ← File readers, SQLite, HTTP
└─────────────────────────────────────────────┘
```

This means the business logic never imports concrete file readers or database drivers — it depends on protocol interfaces defined in `ports/`. Tests can swap in mock adapters without touching application code.

## Frontend architecture

```
src/
├── app/              # Next.js App Router pages (12 routes)
├── components/       # React components by feature domain
│   ├── analytics/    # Charts, KPI cards, cost breakdowns
│   ├── conversation/ # Message cards, DAG views, evaluations
│   ├── databases/    # Database dashboard, OLAP preview
│   ├── evaluations/  # Prompt manager
│   ├── layout/       # Sidebar, breadcrumbs, page header
│   ├── plans/        # Plan viewer, plan panel
│   └── ui/           # shadcn/ui primitives (Card, Badge, Tabs, etc.)
├── hooks/            # SWR data-fetching hooks (20+ hooks)
└── lib/              # Utilities, API client, types, pricing
    ├── client/       # HTTP client, endpoints, normalizer, mutations
    ├── types.ts      # Core TypeScript interfaces
    ├── pricing.ts    # Token cost calculation
    └── constants.ts  # Model pricing tables
```

The codebase is migrating to an explicit layer structure — new code should use:
- `src/features/*` — user-facing feature components and hooks
- `src/views/*` — page-level composition components
- `src/shared/*` — reusable primitives (`shared/ui/`, `shared/layout/`)

Legacy `src/components/*` and `src/hooks/*` paths remain as compatibility shims during the migration.

## Key design decisions

- **Local-first** — zero data transmission, all processing on your machine
- **Streaming parser** — handles 200MB+ JSONL files without loading into memory
- **SWR caching** — conversations refresh every 3s, analytics every 15s, DB status every 5 min
- **Dual database** — SQLite for transactional reads, DuckDB for analytical queries
- **Normalization layer** — `normalize.ts` bridges the snake_case backend and camelCase frontend during the ongoing migration
- **Provider abstraction** — Claude and Codex share the same data model and UI, differing only at the parser level
