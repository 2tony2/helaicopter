## Storage overview

Helaicopter uses a multi-database architecture where each store is optimized for its workload:

| Store | Engine | Role | Location |
|-------|--------|------|----------|
| OLTP | SQLite | Operational metadata, conversations, evaluations | `database-artifacts/oltp/helaicopter_oltp.sqlite` |
| OLAP | DuckDB | Analytics warehouse, time-series queries | `database-artifacts/olap/helaicopter_olap.duckdb` |
| Prefect | PostgreSQL | Orchestration control plane (external) | Configured via Prefect |
| Backend cache | In-process | LRU cache for hot reads | Memory |
| Frontend cache | SWR | Client-side response cache | Browser memory |

## OLTP database (SQLite)

The primary operational store. Contains:

| Table | Purpose |
|-------|---------|
| `conversations` | Persisted conversation summaries with token/cost stats |
| `conversation_messages` | Individual message records |
| `message_blocks` | Content blocks (text, tool calls, thinking) |
| `conversation_plans` | Plan step records |
| `conversation_tasks` | Task records per session |
| `conversation_subagents` | Subagent parent-child relationships |
| `conversation_evaluations` | LLM evaluation execution records |
| `evaluation_prompts` | User-defined evaluation prompt templates |
| `subscription_settings` | Claude Pro/Max/Team plan configuration |
| `refresh_runs` | Database refresh execution log |
| `context_buckets` | Per-tool token attribution |
| `context_steps` | Per-step token attribution |

Managed by SQLAlchemy 2 ORM with Alembic migrations.

## OLAP database (DuckDB)

The analytics warehouse. Materialized during the refresh pipeline for fast aggregation queries. Used by the analytics endpoints for time-series data, model breakdowns, and cost calculations.

DuckDB's columnar storage is ideal for the `GROUP BY` and windowed aggregation queries that power the analytics dashboard.

## Refresh pipeline

The refresh pipeline re-parses conversation source files and updates both databases:

```
~/.claude/ ─┐                    ┌─▶ SQLite (OLTP)
             ├─▶ JSONL Parser ──┤
~/.codex/ ──┘                    └─▶ DuckDB (OLAP)
                                           │
                                           ▼
                                  Schema docs generated
                                  Backend caches cleared
```

### Triggering a refresh

**From the UI:**
Click "Refresh Overview" on the `/databases` page.

**From the API:**
```
POST /databases/refresh
```

**From the CLI:**
```bash
npm run db:refresh
```

### What happens during refresh

1. Scans `~/.claude/` and `~/.codex/` for conversation JSONL files
2. Streaming parser processes each file (handles 200MB+ files)
3. Extracts messages, tool calls, token counts, plans, tasks, subagent relationships
4. Detects changed and deleted conversations using per-conversation refresh fingerprints
5. Upserts only changed conversation detail into SQLite and removes deleted conversations
6. Updates DuckDB facts without dropping the warehouse file as the normal path
7. Rebuilds derived aggregate tables from the current conversation set
8. Invalidates only backend read-cache keys tied to database-backed analytics and conversation lookups
9. Disposes and recreates SQLAlchemy engine (fresh connections)
10. Generates schema documentation under `public/database-schemas/`

The normal near-real-time path is incremental. A destructive full reconciliation is reserved for explicit `force` refreshes or first-time bootstrapping.

### Database migrations

After pulling changes that modify the schema:

```bash
npm run db:migrate:oltp   # Run SQLite (Alembic) migrations
npm run db:migrate:olap   # Run DuckDB migrations
```

## Caching layers

### Backend in-process cache

A lightweight key/value cache in `bootstrap/services.py`. Caches frequently-read data like conversation summaries to avoid repeated file parsing. Database refresh invalidates the database-backed read keys (`analytics`, Codex artifact maps, and database-status entries) instead of wiping unrelated cache entries.

The SQLAlchemy engine is also disposed on refresh, ensuring new requests get fresh database connections.

## Near-real-time analytics supplement

Analytics reads use the warehouse as the authoritative source for stable history. For the newest slice, `/analytics` also checks SQLite conversation summaries for a bounded 6-hour supplement window so newly changed sessions can appear before the next warehouse refresh finishes.

- Warehouse rows remain authoritative outside the 6-hour window.
- Inside the 6-hour window, SQLite rows replace warehouse rows with the same `conversation_id` to avoid double counting.
- SQLite-only recent conversations are included so brand-new sessions appear during near-real-time polling.

### Frontend SWR cache

The frontend uses SWR's built-in caching with per-hook refresh intervals:

| Data type | Refresh interval |
|-----------|-----------------|
| Conversations | 3 seconds |
| Analytics | 15 seconds |
| Database status | 5 minutes |
| Orchestration | SWR default |

This means the UI stays reasonably fresh without manual refreshing, while keeping network traffic manageable.

## Monitoring database health

The `/databases` page provides:

- **Refresh status** — last refresh time, success/failure, trigger button
- **Per-database health** — connection status, table count, size
- **Table inventory** — schema preview with columns, types, and row counts
- **Schema docs** — generated HTML documentation under `public/database-schemas/`

API equivalent:
```
GET /databases/status
```
